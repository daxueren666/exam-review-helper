📚 exam-review-helper 优化说明

用大白话讲清楚每个优化是什么、为什么、怎么做的

1多格式支持 新功能
以前的问题：只能读 PDF。学生有 Word 讲义、TXT 笔记、Markdown 教材就用不了。
现在的改进：支持 4 种格式——PDF、Word（.docx）、纯文本（.txt）、Markdown（.md）。上传什么都能处理。
大白话：以前只能读"印刷书"（PDF），现在手写笔记（TXT）、电子讲义（Word）、网课资料（Markdown）都能读了。

怎么实现的：
  • PDF 用 docling 库（带 OCR，能识别扫描版图片）
  • Word 用 markitdown 库（微软 2026 年新出的，内部调 mammoth）
  • TXT/Markdown 用 charset-normalizer 库（自动检测编码，不会乱码）
踩过的坑：markitdown 对 TXT 不检测编码，中文 GBK 会乱码——所以 TXT/MD 单独用 charset-normalizer。
2文科/理科自动识别 新功能
以前的问题：只针对理工科优化——要求每章至少 10 个概念、必须提取公式、易错点必须是计算错误。文科教材（如毛泽东思想概论）用这套规则完全不合适。
现在的改进：自动识别教材是文科还是理科，用不同的提取规则。用户不用选，上传就行。
大白话：以前只有"理科试卷"（要公式、要计算），现在自动判断：
  • 看到数学公式、∑∫√ 符号 → 当理科处理（重公式、重计算）
  • 看到"观点、理论、意义、历史地位" → 当文科处理（重观点、重论述）
具体区别：
维度理科文科
每章概念数≥10≥8
公式必须提取可选（多数没有）
易错点计算错误概念辨析（如"毛泽东思想≠毛泽东晚年错误"）
重点公式推导核心观点+理论意义

怎么实现的：新建了 system_reader_liberal_arts.md（文科版提取指令）和 system_verifier_liberal_arts.md（文科版核查指令）。宿主读完教材自动判断用哪套。
3图片剥离 优化
以前的问题：Word 文档里的图片会被转成一长串 base64 代码塞进笔记里。毛泽东思想概论 37 页提取出来 590KB，大部分是图片编码，笔记臃肿，AI 读起来也慢。
现在的改进：加 --strip-images 选项，图片存成独立文件（images/img_001.png），笔记里只放一个引用链接。
大白话：以前把每张照片都打印塞进笔记本，本子鼓鼓囊囊。现在照片单独存相册，笔记本里只写"见相册第1张"。

怎么实现的：
  • 扫描 markdown 里的 ![](data:image/jpeg;base64,...)
  • 把 base64 解码成图片文件，存到 images/ 目录
  • 替换为 ![](images/img_001.png)
额外发现：markitdown 其实把图片截断成了 base64... 占位符（没有真实数据），所以还加了清理这些无用占位符的逻辑。
4断点续传 优化
以前的问题：270 页大教材跑 5-pass 要 2 小时。跑到第 8 段中断了（网络断了/电脑关了），下次得从头再来。
现在的改进：每完成一段就存档（.checkpoint/ 目录）。中断后重跑，自动跳过已完成的段，从断点继续。
大白话：就像游戏存档——打到第 8 关存档，死了不用从第 1 关重来，从第 8 关继续。

怎么实现的：
  • Pass 1 完成后存 .checkpoint/segmentation_map.json（分段方案）
  • Pass 2 每完成一段存 .checkpoint/notes_seg-001.json（第1段笔记）
  • 重跑时 run_pipeline.py 检查这些文件，跳过已完成的
用户不用做任何事，脚本自动检测。
5OCR 纠错 优化
以前的问题：扫描版 PDF（拍照版教材）经过 OCR 识别后，常见汉字会认错——"已"认成"巳"、"未"认成"末"、"千"认成"干"。读起来别扭，AI 提取知识点也会被误导。
现在的改进：加 --ocr-correct 选项，自动修正常见 OCR 误判。
大白话：扫描版教材就像拍照识字，会认错字。这个功能相当于一个"校对员"，自动把"巳知"改回"已知"。

怎么实现的：
  • 建了一个常见 OCR 错误字典（巳→已、末→未 等）
  • 上下文校验：数字上下文的"干"→"千"（如"5干3百"→"5千3百"）
  • 保守纠错——宁可漏改不可错改（错改比不改更误事）
只对扫描版 PDF 有效，原生电子版 PDF 不需要。
6公式增强 优化
以前的问题：docling 提取 PDF 时，公式显示为 <!-- formula-not-decoded -->（公式未解码）。AI 不知道哪页有公式、有几个，只能盲猜重建。
现在的改进：新增 enhance_formulas.py 脚本，扫描所有公式占位符，告诉 AI 哪页有公式、有几个。可选装 pix2tex 做公式 OCR。
大白话：以前 AI 看 PDF 笔记，公式位置都是空白（"这里有个公式但没识别"）。现在给它一份清单："第3页有2个公式、第5页有1个公式"，AI 提取时心里有数。

怎么实现的：
  • 扫描 markdown 里所有 <!-- formula-not-decoded -->
  • 按页码统计，输出 formula_hints.json
  • 可选：装 pix2tex 后，把 PDF 页面渲染成图片，用 AI 识别公式转 LaTeX
默认不需要装 pix2tex——AI 基于上下文重建公式已经够用（实测同济高数 73 个公式全部成功重建）。
75-pass 自动化 优化
以前的问题：跑完整流程要手动一步步来：先 extract，再手动读 prompts 跑 5-pass，最后再 generate。容易漏步骤。
现在的改进：新增 run_pipeline.py，一条命令编排全流程——自动提取、自动识别文理科、打印 5-pass 指引、检测断点、最后自动生成 HTML。
大白话：以前要手动切菜、开火、翻炒、装盘。现在按一个按钮，它告诉你"先切菜（Pass 1）→ 再开火（Pass 2）→ ..."，做完一步自动进入下一步。

怎么实现的：
  • Phase 0：调 controller.py extract 提取文档
  • 自动检测模式（文科/理科）
  • 检查 .checkpoint/ 断点
  • 打印详细的 Pass 1-5 指引（读哪个 prompt、输出什么）
  • 检测到 knowledge.json 后自动生成 HTML
用法： 
python scripts/run_pipeline.py textbook.pdf           # 开始流程
python scripts/run_pipeline.py textbook.pdf --generate-only  # 5-pass 完了，生成 HTML
8代码质量优化 优化
除了功能，还做了一轮代码审查和重构：
  • 去重：extractors.py 三个提取函数 90% 重复，提取共享 helper _extract_text_based
  • 修 bug：同名不同格式文件（a.docx + a.md）合并时互相覆盖 → 子目录改为 stem (ext)
  • 文档同步：README 从"17 个测试"更新到"74 个测试"，补全多格式说明
  • 测试覆盖：从 0 个测试 → 74 个单元测试（含 PDF 回归、GBK 编码、OCR 纠错、流水线等）
  • eval 完善：从 3 个 eval → 9 个（含混合格式、负向触发——小说总结不应该触发）
  • 目录修正：html-template.md 的 TOC 行号过时（说 1493 行实际 2507 行）→ 已修正

📊 总结

支持格式 PDF + Word + TXT + Markdown（4 种）
教材类型 理工科 + 文科（自动识别）
单元测试 74 个，全部通过
eval 用例 9 个（含负向触发）
脚本数 6 个（controller / extractors / generate_html / chapter_selector / run_pipeline / enhance_formulas / ocr_corrector）
用户操作 只需上传教材，其余全自动
